# Ecosystem Routing

The skill must run without optional dependencies.

## Discovery Rule

Before invoking any named tool, organ, connector, CLI, or skill:

1. Inspect the current runtime or registry.
2. Confirm the exact callable name and schema.
3. Invoke only what is actually available.
4. State when an expected capability is absent.

Do not infer tool existence from repository names, old documentation,
prior conversations, or architecture diagrams.

## Optional Routing

When available and relevant:

- use current web search and official sources for public facts;
- use dedicated market-data functions for current prices;
- use WEALTH functions for capital mechanics, not for source verification;
- use GEOX functions for geological or physical-resource interpretation;
- use WELL functions for human readiness or organisational strain;
- use arifOS Kernel for actual routing or judgment only when invoked and returned by the live tool;
- use Hugging Face or vendor tooling for model, dataset, license, and repository evidence.

If these are unavailable, continue using public evidence and label the
missing analytical layer.

## Authority Boundary

Tool output is evidence from that tool, not automatic sovereign truth.
Do not upgrade a computation, model opinion, or retrieval result into a SEAL.

Keep these separate:

- **content audit**: whether the briefing claims are accurate;
- **runtime audit**: whether tools, authority, registry, or transport are coherent;
- **decision**: what Arif chooses to do.

Combine them only when the user's request explicitly requires a joint assessment.
