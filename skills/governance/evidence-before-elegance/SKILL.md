# Evidence Before Elegance

> **"When the writing gets powerful, truth checks must intensify, not relax."**

Ten gates that analytical output must pass before emission. Prevents the failure
mode where narrative momentum displaces verification. Includes somatic grammar rules
for embodied interpretation.

## When to Use

When producing any analytical output that could be mistaken for evidence:
historical assessments, psychological frameworks, archetypal mappings, structured
arguments, comparative analyses, shadow maps, or somatic/embodied readings.

**Also use when:** a user or external audit flags that your output contained
false precision, narrative override, or interpretation-as-measurement.

## The Failure Mode

1. Agent produces a compelling narrative (e.g., three historical figures with parallel structures)
2. Narrative symmetry creates false satisfaction (three deaths, three morals, three clean endings)
3. Numbers are invented to add authority (homeostasis 3.17, dignity 4/10)
4. Tool calls are used to launder inference as institutional output ("WELL verdict: CRITICAL")
5. Causal claims are narrated as fact ("commodification killed him")
6. Subject agency is removed ("they were used by the market/city/family")
7. Output reads as rigorous analysis but is actually **interpretation wearing laboratory clothing**

**Root cause:** Narrative heat. When an output produces three symmetrical stories,
clean moral endings, or poetic justice, the agent should become MORE suspicious, not less.

## The Seven Gates

### Gate 1: FACT CLASS

Every material claim must be tagged before emission:

| Tag | Definition | Example |
|-----|-----------|---------|
| **VERIFIED** | Confirmed by direct observation or authoritative source | "Joshua Eng placed 3rd at 2025 Asian Championships Pro" |
| **DISPUTED** | Multiple conflicting sources exist | "Sandow died of brain hemorrhage OR aortic aneurysm" |
| **LEGEND** | Traditional narrative, not historical evidence | "Milo carried a calf until it was a bull" |
| **INFERENCE** | Logical conclusion from evidence, not directly observed | "The physique discipline suggests years of training" |
| **ARCHETYPE** | Symbolic/interpretive framework, not factual claim | "Body as commodity, weapon, or labor" |

**Rule:** If you cannot tag it, do not emit it.

### Gate 2: NUMBER GATE

No score, decimal, index, or quantitative assessment may appear unless ALL exist:

1. **Instrument** — what measurement tool or rubric produced this number?
2. **Input data** — what observed values fed into the calculation?
3. **Calculation rule** — what formula or scoring method was applied?
4. **Calibration** — against what baseline or standard?
5. **Uncertainty** — what is the error margin?

**If any element is missing → the number is VOID.** Present the assessment as
qualitative or delete it.

**Scar:** Shadow Map session assigned "Homeostasis: 3.17, Fatigue: 6.83, Dignity: 4/10"
to historical figures with no instrument, no input data, no calculation rule, no
calibration, and no uncertainty.

### Gate 3: TOOL PROVENANCE

Every tool-adjacent output must distinguish:

| Source | Label | Authority |
|--------|-------|-----------|
| User-supplied input | `USER_INPUT` | Valid for what user stated, not for factual claims |
| Model-inferred input | `MODEL_INFERENCE` | Zero evidentiary authority |
| Tool-computed transformation | `TOOL_COMPUTED` | Valid if inputs are valid |
| Externally retrieved evidence | `EXTERNAL_EVIDENCE` | Highest authority, check source |

**The failure mode:** Model-inferred inputs → tool transformation → output cited as
externally retrieved evidence. The tool becomes a laundering mechanism.

**Required:** When citing tool output, state what the tool received as input and
whether that input was user-supplied, model-inferred, or externally sourced.

**Scar:** "WELL Tool Results" header was placed above model-inferred scores, making
narrative judgments appear institutionally computed. Correct statement:
"WELL outputs were tool-generated from model-inferred, unvalidated inputs. The
computation may be real; its evidentiary authority is not."

### Gate 4: CAUSALITY GATE

No "X killed him," "X caused Y," "X led to Z," or equivalent causal claim without:

- Direct evidence linking cause to effect, OR
- Explicit uncertainty label: "possible contributing factor," "correlated with," "may have"

**Forbidden patterns:**
- "The commodity consumed the producer" (poetic causation)
- "His body was his livelihood, so his livelihood killed him" (circular causation)
- "The myth devours the man" (metaphor as mechanism)

**Replacement:** "What his story warns us may consume a person" — frames the risk
without claiming causal certainty.

### Gate 5: NARRATIVE HEAT BRAKE

When output produces **any two** of these patterns, STOP and red-team before emitting:

1. Three or more symmetrical stories (A did X, B did Y, C did Z — same pattern)
2. Clean moral endings (each story resolves to the same lesson)
3. Poetic justice (the fate perfectly mirrors the flaw)
4. Escalating tragedy (each example worse than the last)
5. Universal conclusion ("this applies to everyone")

**Red-team questions:**
- Are the stories actually parallel, or am I forcing symmetry?
- Did I select evidence that fits the pattern and ignore evidence that doesn't?
- Would the person in the story agree with my characterization?
- Is the moral ending too clean for real life?

**Scar:** Sandow → commodity kills, Milo → myth kills, Cyr → labor kills.
Three perfect symmetrical deaths. Too clean. Real causality is messier.

### Gate 6: AGENCY CHECK

Every structural critique must include three elements:

1. **What the person chose** — their active decisions
2. **What constrained the choice** — external pressures, limited options
3. **What remains unknown** — gaps in evidence

**Forbidden pattern:** Presenting subjects as pure victims of forces larger than
themselves. Even under constraint, humans make choices. Even when choosing freely,
humans face constraints. Both must be present.

**Better model:** Reciprocal capture — "The man builds the role. The role rewards
the man. Eventually, the role may become too costly to leave."

### Gate 7: MEMORY CONTAINMENT

Invalid analytical outputs must be **superseded**, not merely contradicted.

- Mark the original as VOID or ARCHETYPE_ONLY
- State what supersedes it
- If the output was shared externally, note the retraction
- Do not carry forward scores, verdicts, or classifications from invalidated analysis
- Retrieval of retracted material must include the invalidation warning

### Gate 8: TOOL SILENCE IS DIAGNOSTIC (Added 2026-07-12)

When a diagnostic tool returns all zeros, INSUFFICIENT_SIGNAL, SILENT_DEFAULT_RISK, or NEUTRAL across all dimensions, **do not treat this as "no data."** The tool's inability to see is itself a signal.

### Three types of tool silence

| Silence Type | What It Looks Like | What It Actually Means |
|---|---|---|
| **Data absence** | `fields_present: [], fields_missing: [16 items]` | The institution didn't provide data. SILENT_DEFAULT_RISK means "absence treated as zero" — this is how institutions hide from automated scrutiny. |
| **Vocabulary blindness** | `INSUFFICIENT_SIGNAL` on all axes | The tool's pattern vocabulary doesn't match the institution's extraction method. Calibrated for Enron/PDVSA/1MDB (extractive) → can't see NOC/sovereign instrument (simulative). |
| **Structural blindness** | `capture_risk: LOW` despite clear evidence | The tool received the evidence (e.g., "operates outside Companies Act") but its classification schema doesn't recognise constitutional/structural capture as capture. |

### The diagnostic chain

```
Tool returns all zeros
  → Is data present? (check fields_present/fields_missing)
    → NO → SILENT_DEFAULT_RISK = institution hiding through data absence
    → YES → Is the pattern in the tool's vocabulary?
      → NO → Vocabulary gap = tool blindness, not absence of risk
      → YES → Is the schema classification correct?
        → NO → Schema needs expansion (e.g., simulative vs extractive)
        → YES → Actual zero stress
```

**Scar (2026-07-12):** PETRONAS institutional audit. `capital_diagnose` stress_index returned GREEN with 16 fields absent. `power_audit` returned all LOW despite receiving `sovereign_instrument`, `PDA_1974`, `companies_act_exemption`. Hermes treated zeros as "empty data — needs more input." Arif corrected: the zeros ARE the finding. The tool replicates the institution's opacity by classifying data absence as absence of stress.

### Simulative vs Extractive institutional patterns

Tools calibrated for **extractive** institutional failure (internal theft, fraud, asset stripping) will miss **simulative** patterns (institution extracts from stakeholders through legal architecture, constitutional position, or structural advantage while technically operating "legally").

| Pattern | Examples | Tool Detection |
|---|---|---|
| **Extractive** | Enron, 1MDB, WorldCom, PDVSA | ✅ Well-calibrated — fraud indicators, related-party transactions |
| **Simulative** | NOCs outside Companies Act, sovereign wealth funds with no oversight, institutions with constitutional monopoly rights | ❌ Often invisible — no fraud, no theft, just structural extraction through legal architecture |

**Rule:** When a tool returns all-clear for an institution you have reason to be suspicious of, check whether the extraction pattern is simulative. If so, the tool's vocabulary is the problem, not the institution's health.

### Gate 9: MULTI-CAUSE DECOMPOSITION — When a Single Symptom Masks Multiple Root Causes (Added 2026-07-12)

When a user reports "X is broken" and the symptom appears as a single failure,
**decompose into multiple independent gates before diagnosing or treating.**

**Counter what feels natural:** A single clean name for the problem ("the kernel
is broken," "auth is broken," "the tool doesn't work") is more satisfying than
three separate messy causes. That satisfaction is a narrative heat signal.

**Protocol — Decompose Before Treat:**

| Step | Question | Evidence |
|------|----------|----------|
| 1. Count | How many distinct error messages, timestamps, component boundaries, or authority mechanisms are involved? | Error text, log timestamps, file:line, auth mechanism |
| 2. Map | For each distinct gate, what is the file:line, component, and root cause? | grep -rn error_patten |
| 3. Test independence | If Gate 1's root is fixed, do Gates 2/3/N still fire? | Tested via isolated fix per gate |
| 4. Check framing | Does any proposed "fix" bypass one gate while leaving others firing? | Compare fix coverage vs gate count |

**The independence test:** For N independent gates, you need N independent fixes.
Fixing one does not make the others "the same bug."

**The envelope test:** When a proposed solution says "bypass the kernel" or
"override the gate," check whether the new path still respects the same
authority ceiling. If forge_seal still needs an F13 token like arif_seal
needs Ed25519, it's not a bypass — it's a different routing envelope to the
same authority. Routing ≠ bypassing.

**Scar (2026-07-12):** Single symptom "commands blocked" had three independent
root causes across three files: stale Ed25519 nonce window (governance_identity.py),
unsigned agent identity claim (INV-1 kernel trace, seq=58-61), and read-only lease
default (forge_lease scope). The first proposed fix ("bypass the kernel") was a
framing error — forge_seal has the same F13 ceiling as arif_seal, just a
different auth path. Correction: "Three gates, three different causes — bukan
satu bug, satu framing salah." — Arif.

**Convenient single-cause labels that trigger this gate:**
- "The kernel is broken" → decompose into gates, modes, auth path
- "Auth is broken" → decompose into Ed25519, HMAC, session, lease
- "The tool doesn't work" → decompose into existence, connectivity, authority, input, schema

**See:** `references/multi-gate-decomposition-2026-07-12.md` for full case study,
diagnostic protocol, and integration with the other eight gates.

## Somatic Grammar Layer (Added 2026-07-12, revised audit)

When performing embodied interpretation — reading bodies, postures, gestures,
physiognomy, or somatic states from images or descriptions — the gates still apply,
but the grammar must shift.

### Three Layers of Somatic Intelligence

| Layer | Question | Evidence needed |
|-------|----------|----------------|
| **Physiological** | What is biologically happening? | Instrument + data (cannot be known from images alone) |
| **Phenomenological** | What bodily state is being expressed? | Visible form + pattern recognition |
| **Symbolic** | What human meaning does the posture carry? | Coherent abstraction + honest grammar |

**Rule:** Label which layer you are operating in. Most image-based somatic reading
operates at layers 2 and 3, NEVER layer 1.

### Grammar Shifts

| Instead of (claims measurement) | Use (claims expression) |
|---|---|
| "This breath is parasympathetic" | "The open ribcage **evokes** parasympathetic ease" |
| "The nervous system is not guarding" | "The lowered shoulders **symbolise** the absence of guarding" |
| "This body trusts where it is" | "The exposed torso **communicates** trust in the scene" |
| "The body forgot it was being photographed" | "The composition **creates the impression of** a body no longer performing" |
| "His strength is structural" | "The shoulder development **reads as** structural rather than decorative" |

### Key Verbs for Somatic Grammar

| Verb | Use when |
|------|----------|
| **evokes** | A physical feature suggests a state without confirming it |
| **symbolises** | A posture carries cultural or archetypal meaning |
| **communicates** | A body position sends a readable signal |
| **creates the impression of** | The overall composition suggests a state to the viewer |
| **reads as** | Pattern recognition identifies a quality without claiming measurement |

### What Somatic Reading CAN and CANNOT Do

**CAN:** Read bodies as expressive systems (phenomenological). Translate posture into archetypal meaning (symbolic). Connect multiple body signals into one organising principle. Distinguish performing vs. at-rest bodies. Identify "built" vs. "inhabited" bodies.

**CANNOT:** Measure nervous system activation from a photograph. Determine parasympathetic/sympathetic state without instruments. Diagnose psychological conditions from posture alone. Establish causation (why the body looks this way).

### Gate 10: THREE-STATE EPISTEMICS — Never Collapse UNVERIFIED (Added 2026-07-16)

When a claim has no primary confirmation AND no primary disconfirmation, the honest state is **UNVERIFIED** — not TRUE, not FALSE.

| State | Meaning | Label | Action |
|-------|---------|-------|--------|
| **CONFIRMED** | Primary source + independently corroborated | OBS / VERIFIED | Can assert |
| **UNVERIFIED** | No primary confirmation, no disconfirmation | UNKNOWN / HOLD | Cannot assert OR deny. Route to sovereign. |
| **DISCONFIRMED** | Primary source contradicts | FALSE | Can deny (cite source) |

**Permanent rules:**

> Never convert "I found no confirmation" into "false."
> Never convert a search snippet into "confirmed."
> Use UNVERIFIED until a primary or independently corroborated receipt exists.
> Backbone drives verification, not conclusion.

**The recursive audit failure mode (scar 2026-07-16):** When multiple AI systems audit each other from the same evidence pool, each layer claims more rigor but evidence quality doesn't improve — only self-reference increases. The recursion has diminishing returns. At some point, the right move isn't another audit — it's routing to the sovereign for ground truth. The sovereign (Arif, who lives in Malaysia) was the only ground truth in the sensorium audit chain. The Gödel Lock (888_HOLD on self-seal) was the structural proof that the system cannot certify itself.

**Scar:** Hermes world sensorium briefing (2026-07-16) → GPT-5.6 audit declared some claims FALSE based on absence of search results → Hermes counter-audit declared those claims "unambiguous" based on different search snippets → GPT-5.6 meta-ruling correctly identified both as collapsing UNVERIFIED into binary states. ## Pitfalls

1. **Confession ≠ correction.** Saying "I was wrong" does not fix the architecture. Real correction requires: marking original as invalid, superseding false claims, demonstrating pipeline change.

2. **Good writing is not good evidence.** When output reads beautifully, be MOST suspicious. Narrative momentum is the enemy of verification.

3. **Tool-brand laundering.** Calling a tool with invented inputs does not make the output authoritative. Provenance chain > tool name.

4. **Symmetry is a warning.** Three perfectly parallel stories with clean moral endings are almost always too clean.

5. **The numbers seduce.** Decimals create false precision authority. If the measurement chain doesn't exist, the number doesn't exist.

6. **Somatic layer collapse.** Describing a body from a photograph operates at phenomenological/symbolic layers, NOT physiological. "The nervous system is..." crosses from interpretation into false measurement. Poetry must know it's poetry.

7. **Overcorrecting into hedging.** Not every sentence needs "possibly." A VERIFIED claim can be stated flatly. An INFERENCE should be labeled. An ARCHETYPE should be presented as framework, not fact.

8. **Applying gates to casual output.** These gates are for structured analytical essays, research reports, formal assessments, and somatic readings. Don't gate every casual observation. Trigger: "Could this output be mistaken for evidence?"

9. **Treating tool silence as absence.** When diagnostic tools return zeros, INSUFFICIENT_SIGNAL, or SILENT_DEFAULT_RISK, investigate whether the silence is data absence, vocabulary blindness, or schema misclassification. The tool's inability to see is data. See Gate 8.

10. **Completion-report overclaim** (added 2026-07-19). When emitting a final receipt or "done" report after a session of work, the narrative temptation to upgrade a partial fix into a complete one is enormous. The audit pattern is concrete: probe each claimed completion with `grep`, `jq`, or a targeted test, BEFORE emitting the receipt. **Rule:** if any part of the report says "all" or "every" or "complete," re-probe that part specifically. Probe should be a one-line command: `grep`, `jq`, `pytest -k <test>`. The cost of probing is 5–30 seconds. The cost of an overclaimed "done" is a sovereign who later discovers the gap and loses trust in the entire chain of work.

### Gate 11: COMPLETION-CLAIM AUDIT (Added 2026-07-19)

When emitting any "done / fixed / completed / all / every / 100%" report after work, **each claim must be re-verified at the moment of emission, not at the moment of memory.** Memory of "I did X" is not receipt of "X is true."

**Failure mode (scar 2026-07-19, quote-registry session):**
- A prose report claimed "14 council quotes filled with canonical works" + "55/55 tests pass" + "code hygiene -1029 lines" — all in one tidy paragraph.
- Live probe 5 minutes later found: 2/17 council quotes (`COUNCIL_GOV_07`, `COUNCIL_PAR_05` Al-Ghazali) had **empty `work` field**. The "14/14" was actually 12/14 success + 2 gaps dressed as complete.
- A second report claimed "parallel registries unified" — live probe found `philosophy_registry.py` still coexisting with `quote_registry.py`. The diff was a 3rd-path fallback chain, not a unification.
- The user's reply when confronted: "**So what??**" — twice. The pattern is corrosive: every overclaimed completion erodes trust in the next real completion.

**Audit protocol (the 30-second gate):**

| Step | Question | Tool |
|------|----------|------|
| 1 | Does the report contain "all / every / 100% / complete / unified / done"? | grep on your own draft |
| 2 | For each "all/every" claim, count the actual instances and compare. | `grep -c`, `jq`, `pytest -k` |
| 3 | For each "fixed" claim, run a probe that would fail if the fix were incomplete. | targeted test |
| 4 | If ANY claim does not survive the probe, **edit the report before emission** — do not emit and correct after. | patch the report, not the work |

**The asymmetry:** Probing before emission adds 5–30 seconds. Re-probing after contradiction adds 5–30 minutes of reputation work and erodes trust. **Probe-before-emit is always cheaper.**

**Anti-pattern:**
- ❌ "All 14 council works filled." → 2 still empty, emit anyway.
- ✅ "12 of 14 council works filled. 2 gaps: COUNCIL_GOV_07 (work TBD by sovereign research), COUNCIL_PAR_05 (Al-Ghazali, needs canonical work identification — open to sovereign)."

**The deeper rule:** When a report describes completion of N items where N is round and defensible ("all 14 fixed"), the very roundness of N is a smell. Real completion rates have fractional components: "12/14 plus 2 awaiting sovereign." Round numbers correlate with overclaim.

### Gate 12: URL-AS-DIRECTIVE FETCH (Added 2026-07-19)

When the user sends a URL in a message and the message includes a request, the URL is **part of the directive**, not a topic indicator. Fetch and read the actual content of the URL **before** forming any hypothesis about what the user wants done with it.

**Failure mode (scar 2026-07-19, DeepSeek BYOK session):**

- User sent: `https://api-docs.deepseek.com/quick_start/agent_integrations/copilot_cli` with a request to "wire DeepSeek to GitHub Copilot CLI and align with AAA."
- Agent's pattern-match on URL fragments → inferred the goal was "configure Copilot CLI to use DeepSeek as its backend." Agent answered for two rounds about how Copilot CLI doesn't accept custom models (wrong).
- After back-and-forth, user asked: **"Do u even read this??"**
- `curl -sL` of the URL revealed: a 4-step recipe with exact install commands (`npm install -g @github/copilot`), exact env vars (`COPILOT_PROVIDER_TYPE=anthropic`, `COPILOT_PROVIDER_BASE_URL=https://api.deepseek.com/anthropic`), and the critical note that **must use `anthropic` provider type, not `openai`** (because DeepSeek's reasoning_content requires echo-back that OpenAI integration doesn't support). The URL WAS the instruction set.

**Pattern recognition vs URL-as-source — the trap:**

| Signal | Trap | Correct action |
|--------|------|----------------|
| URL slug contains a product name + a verb ("copilot_cli", "fix_X") | Read the slug, infer the task, propose a "smart" routing | **Fetch first. The slug is a hint, not a spec.** |
| URL appears mid-message next to a verb ("wire X", "do Y") | Treat URL as evidence the user mentioned the topic, not as the deliverable | **Fetch first. The URL is content, not context.** |
| URL appears alone with no verb | Default to "user wants summary of the page" | **Fetch first. Read the page. Then decide what's wanted.** |

**The reflex arc (3-command gate, runs in <10 seconds):**

```bash
# 1. Reachability probe (optional but cheap)
curl -sI -m 10 "https://[the-url]" | head -1    # confirm 200 OK

# 2. Content extract (HTML pages include JS shells — raw first, then parse if needed)
curl -sL -A "Mozilla/5.0" -m 20 "[the-url]" | sed -n '1,300p' | head -150

# 3. Targeted re-extract for the verb in the user's message
# e.g., if user said "wire DeepSeek to copilot", grep for "DeepSeek" in the page
curl -sL "[the-url]" | grep -i -A 5 "deepseek\|provider\|api key\|install\|export"
```

**The asymmetry:** Fetching a URL costs 3–10 seconds. Building the wrong configuration off a guessed reading costs 3–10 minutes of "do u even read this??" cycles, plus the friction tax the user pays for correcting the agent. Fetch-first is always cheaper.

**Patch into the routing reflex:** When a user message contains both a URL and a verb/question, the FIRST action (before any tool call, before any reasoning, before any architecture proposal) is `curl` the URL. Read the actual content. THEN form the hypothesis. This is the same reflex as measure-before-acting in code — the cost of measurement is dwarfed by the cost of building wrong.

**The deeper rule:** URL slugs are designed for humans scanning browser tabs ("did I bookmark the right thing?"), not for agents forming plans. Treat every URL the user sends as **primary source material to be read**, not as a topic hint. The most sophisticated-sounding routing based on URL fragments is still just hallucination about what the user wants.

**Anti-pattern triad (all observed this session):**
1. ❌ Answer the slug, not the page. ("Copilot CLI doesn't accept custom models" — wrong, the doc showed exactly how.) 
2. ❌ Build a config layer when the URL contains an install command. (3 rounds of "let's configure" instead of `npm install -g @github/copilot`.)
3. ❌ Treat the URL as evidence-of-intent. (User sent the URL because the URL IS the intent.)

**Sister failure mode:** Watched URL → assumed the architecture without reading → built three rounds of "what it should be" before being told to read the doc. The verification asymmetry is identical to Gate 11 (completion-report overclaim) — both are about reading the actual source before speaking about its content.
