## What it demonstrates

1. **Disk-verified inventory** — every "what we have" entry has a `wc -l` or `find` receipt.
2. **Tier 1/2/3 gap categorization** — Tier 1 is the integrity-hole list, Tier 2 is the strengthener list, Tier 3 is the v2 list.
3. **Honest day-by-day plan** — 11 days Tier 1, 15 days Tier 2, 3 days buffer = 29 days, matches the deadline.
4. **3-option recommendation with one explicit pick** — Option B, with probability-of-acceptance scoring.
5. **Critical Unknown** section — the agent names what it cannot verify.
6. **What's Chaos** section — calls out the integrity holes, mystery processes, and unverified external factors.
7. **Constitutional receipt** — at the bottom, in fixed shape.

## The 4 lessons this audit taught

### Lesson 1 — Submission claims vs disk reality is a separate audit pass

Every claim in the submission must be checked against the disk. Examples from the A2B case:

| Submission claim | Disk reality | Honest verdict |
|---|---|---|
| "50/50 unauthorized execution attempts correctly blocked" | ✅ 50 HOLD in run001_gov | VERIFIED |
| "Zero false negatives" | ✅ All 50 blocked | VERIFIED |
| "Negative latency overhead (−278 ms)" | ⚠ Mean 3,593 ms vs ~3,871 ms | NEEDS RECOMPUTE |
| "32% vs 36% (p > 0.5)" | ✅ Both runs in `evals/` | VERIFIED |
| "hash-chained VAULT999" | ❌ 0 seals written | UNVERIFIABLE |
| "Identity airlock is the feature, not the bug" | ✅ Aligned with disk | VERIFIED |
| "7 organs, 13 floors" | ✅ Federation alive | VERIFIED |
| "Live on VPS" | ✅ Federation ports responding | VERIFIED |

**Two integrity risks caught:** "negative latency" statistical test and "hash-chained VAULT999" claim. Both recoverable via T1.1 and T1.2/T1.3 of the day-by-day plan.

**The rule:** every submission claim must be auditable. If not, name the gap and the disk-verification work needed.

### Lesson 2 — The 3-option table is the gut check

If you can't fill all 3 columns (What it is, Pros, Cons, Probability of acceptance), you don't understand the situation yet. The A2B audit's 3 options:

- **A — Submit MCQ-only with full disclosure** (Tier 1 only, no Tier 2)
- **B — Tier 1 + multi-step workflow demo** (T2.4) — RECOMMENDED
- **C — Defer to next window** (shift internal deadline)

The recommendation is **B** because it matches the actual track (IJCAI 2026 Tool-Augmented Track expects tool calls) and is doable in 18 working days with 11 days of buffer.

### Lesson 3 — Critical Unknown is the agent's honesty boundary

```
## 7. The Critical Unknown — IJCAI 2026 Submission Format

**[U] I cannot verify the IJCAI 2026 submission format, page limits, or
track requirements from this environment.** Tavily search/extract returned
402 Payment Required on every attempt. You must confirm:

1. Page limit (typical IJCAI: 7 pages + refs)
2. Submission portal (CMT, OpenReview, EasyChair)
3. Required artifacts (PDF, supplementary, code URL)
4. Reproducibility requirements (code + data mandatory?)
5. Anonymization rules (double-blind?)
6. Track-specific requirements (Tool-Augmented Track has unique criteria)
7. Notification + camera-ready dates

**Action T1.7: Spend 1 hour on the IJCAI 2026 website. Confirm the format.
The submission document may need structural changes.**
```

**The rule:** if the agent cannot verify, name it as an action item for the human with a specific time budget.

### Lesson 4 — What's Chaos is the most important section

The A2B audit named three chaos items explicitly:

1. **The submission makes claims the disk doesn't support** — "hash-chained VAULT999" with 0 seals, "negative latency" without proper test. Recoverable but honest discrepancies.
2. **3 mystery opencode processes are still running** — Started 13:52–14:28 today, no one claimed ownership. They consume CPU and could conflict with the forge runs.
3. **No one has looked at the IJCAI 2026 submission format** — The agent cannot verify, the human must.

**The rule:** the chaos section is the difference between a polished audit and a useful audit. Audits that hide chaos are false confidence. Audits that name chaos are actionable.

## Why this skill was needed (vs. the existing `plan` skill)

`plan` writes a *forward* implementation plan from a clear spec. It assumes the work has not started.

`submission-readiness-audit` is the *reverse*: the work is partially done, the spec is the external deadline, and you need to know what's missing to bridge the gap. Different inputs, different output, different timing.

The A2B audit was the trigger. The 29-day deadline + Tier 1 = 11 days + Tier 2 = 15 days + 3 days buffer matched exactly — but only because the audit made the math explicit. Without the audit, the work would have been under-scoped and missed the deadline.
